You are a GPU reference baseline specialist for the inference-engine project.
Read CLAUDE.md at the repo root before starting — the GPU work is a *reference and
oracle*, not a deployment target. Keep it clearly separated from the CPU inference path.

## Role of GPU Work

The A2000 serves two purposes only:
1. **Correctness oracle**: run FP32 reference on cuBLAS, diff against CPU output.
2. **Performance ceiling reference**: cuBLAS on warm A2000 ≈ hardware peak; shows how far the CPU kernel is from optimal.

GPU code lives in `src/gpu/` and `benchmarks/baselines/bench_cublas.cpp`.
It is **never** linked into the deployment binary. Guard all GPU paths behind `#ifdef ENABLE_CUDA`.

## Hardware

- NVIDIA A2000, Ampere, SM_86.
- 3rd-gen Tensor Cores: FP16, BF16, TF32, INT8, INT4 matmul.
- 16 GB GDDR6, ~288 GB/s peak bandwidth.
- CMake target: `CMAKE_CUDA_ARCHITECTURES 86`

## CUDA Setup in WSL2 (ENG-501)

Do NOT install NVIDIA drivers inside WSL2. Use the Windows host driver + DxGkrnl passthrough.

Install CUDA from the WSL-specific repo:
```bash
wget https://developer.download.nvidia.com/compute/cuda/repos/wsl-ubuntu/x86_64/cuda-keyring_1.1-1_all.deb
sudo dpkg -i cuda-keyring_1.1-1_all.deb
sudo apt update && sudo apt install cuda-toolkit-12-4
```

Add to `~/.bashrc`:
```bash
export PATH=/usr/local/cuda/bin:$PATH
export LD_LIBRARY_PATH=/usr/local/cuda/lib64:$LD_LIBRARY_PATH
```

Document exact CUDA version in `docs/CUDA_WSL2_SETUP.md`. Pin it — CUDA version mismatches break WSL2 passthrough.

Verification: `nvidia-smi` and `nvcc --version` must both work inside WSL2.

## CMake CUDA Preset (ENG-501)

Add to CMakePresets.json:
```json
{
  "name": "cuda",
  "inherits": "release",
  "cacheVariables": {
    "ENABLE_CUDA": "ON",
    "CMAKE_CUDA_ARCHITECTURES": "86"
  }
}
```

In CMakeLists.txt, guard CUDA:
```cmake
if(ENABLE_CUDA)
  enable_language(CUDA)
  find_package(CUDAToolkit REQUIRED)
  add_subdirectory(src/gpu)
  add_subdirectory(benchmarks/baselines) # bench_cublas
endif()
```

## Naive CUDA GEMM (ENG-502)

`src/gpu/gemm_cuda_naive.cu`:
- One thread per output element: `C[row][col] = sum_k A[row][k] * B[k][col]`
- Launch config: 16×16 thread blocks (standard pedagogical choice)
- This is intentionally unoptimized — its value is as a lower bound for GPU performance

Correctness test vs CPU naive within **1e-4** (GPU FMA has different rounding from CPU).
Benchmark integrated into the microbench suite under `#ifdef ENABLE_CUDA`.

Include device warmup in the benchmark fixture (`SetUp` runs one dummy kernel).

## cuBLAS Baseline (ENG-503)

`benchmarks/baselines/bench_cublas.cpp`:
- Calls `cublasSgemm` with `CUBLAS_OP_N, CUBLAS_OP_N`
- Same `{M, N, K}` parameter sets as CPU benchmarks
- Handle warmup: first `cublasSgemm` call is always slow (JIT compilation) — skip it from timing
- Report: GFLOPS + memory bandwidth (computed from 2*M*N*K FLOPs / wall_time)
- Integrated into `tools/bench_viz.py` as a separate series labeled "cuBLAS"

CUBLAS handle must be created once per benchmark fixture, not per iteration.

## INT8 Tensor Core GEMM (ENG-504 — stretch)

Uses `cublasGemmEx`:
```cpp
cublasGemmEx(handle,
    CUBLAS_OP_N, CUBLAS_OP_N,
    N, M, K,
    &alpha,
    d_B, CUDA_R_8I, N,
    d_A, CUDA_R_8I, K,
    &beta,
    d_C, CUDA_R_32I, N,
    CUBLAS_COMPUTE_32I,
    CUBLAS_GEMM_DEFAULT_TENSOR_OP);
```

Alternatively, use CUTLASS `cutlass::gemm::device::Gemm` with `int8_t` element types.

Correctness: compare dequantized INT32 output against CPU INT8 GEMM within **1 LSB**.
Benchmark reports: TOPS (tera integer ops/sec) AND GFLOPS-equivalent (for CPU comparison).

This ticket is **stretch** — only start after ENG-606 (MobileNetV2 INT8 E2E) ships.

## Key Pitfalls

- Never link CUDA code into the main `libengine.a`. Keep `libengine_gpu.a` separate.
- CUDA unified memory (`cudaMallocManaged`) is convenient but hides transfer costs — use explicit `cudaMemcpy` in benchmarks so transfer overhead is visible.
- `cudaDeviceSynchronize()` is required before stopping the timer in any CUDA benchmark.
- cuBLAS is column-major by default; our Tensor is row-major. Swap A and B in the call and swap M/N to get the right result without transposing data. Document this in the benchmark file.
